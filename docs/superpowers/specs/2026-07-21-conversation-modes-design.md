# Conversation Modes — Design Spec

**Date:** 2026-07-21
**Status:** Approved design, pending implementation plan
**Design reference:** claude.ai/design PostAud.io project, `Postaudio Conversation Settings.dc.html` (sections 1a, 1b, 2a–2d)

## Summary

Add three conversation modes to PostAud.io series: **Deep dive** (today's full interview behavior), **Flow** (after each answer the session pauses and offers 2–3 follow-up cards you can answer now, queue for later, or stop on), and **Quickfire** (a preset question list asked one at a time). Modes are a per-series default with an optional per-session picker, backed by a new persistent **question queue**.

Decisions locked during brainstorming:

- Build all of it in one effort (settings + all three mode runtimes + queue + wrap), not phased.
- Mode **replaces** the depth dial in settings. `depth` values migrate: `single` → `quickfire`, all others → `deep`.
- Quickfire's question source is the **queue first, then must-cover topics** (lowest coverage first). No separate authoring list.
- With "Ask me each time" on, the mode picker is a **pre-talk screen**, not an in-session switcher.
- Flow follow-ups are generated by the **realtime model itself via a tool call** (`propose_followups`), not a separate server-side text model.

## 1. Data model

### Migration `0019_conversation_modes.sql`

- `series.conversation_mode` — text/enum `'deep' | 'flow' | 'quickfire'`, NOT NULL, default `'deep'`.
  - Backfill: `UPDATE series SET conversation_mode = CASE WHEN depth = 'single' THEN 'quickfire' ELSE 'deep' END`.
  - `depth` column **stays** (not dropped). It is removed from the settings UI; Deep dive continues to read the stored register (`light`/`balanced`/`deep`) so existing series keep their wording. New series never set it and use `balanced`. A legacy `depth = 'single'` on a `deep`-mode series is treated as `balanced`.
- `series.ask_mode_each_time` — boolean, NOT NULL, default false.
- `interviews.mode` — same enum, nullable; set at session start (default mode or picker choice). Null on historical rows means "pre-modes deep behavior".
- New table `queued_questions`:
  - `id uuid PK`, `series_id uuid FK → series`, `text text NOT NULL`
  - `source text CHECK IN ('flow','member')` — saved from a Flow session vs. manually added
  - `created_by uuid FK → auth.users`, `source_interview_id uuid FK → interviews NULL`
  - `position int NOT NULL` (0 = "Next up"), `status text CHECK IN ('pending','asked','removed') DEFAULT 'pending'`
  - `asked_in_interview_id uuid NULL`, `created_at`, `updated_at`
  - RLS mirrors `topics` (series members read; interview-access members insert; admins update/delete).
- Regenerate `src/db/types.ts`.

## 2. Settings screen (mockup 1a)

`src/app/app/series/[id]/settings/InterviewGuideForm.tsx`:

- **Remove** the depth dial.
- **Add "Default mode"** segmented control — `Deep dive / Flow / Quick fire` — with one-line helper text per mode:
  - Deep dive: "A full guided conversation — the interviewer follows the thread."
  - Flow: "Answer, then choose where to go next. Save follow-ups for later."
  - Quick fire: "One question after another from your queue and topics."
- **Add "Ask me each time"** toggle beneath the segmented control (same card, divider between, per mockup).
- Session length: **unchanged** (existing 10/20/45 values). The mockup's 5/15/Open variant is intentionally not adopted — it would ripple into pacing/billing.
- **Add "Question queue" card**: title, subtitle "Saved follow-ups from Flow sessions", pending count, links to `/app/series/[id]/queue`.
- PATCH persistence via existing partial-diff pattern to `/api/series/[id]` with new fields `conversationMode`, `askModeEachTime` (extend `updateSeriesSchema` and the camelCase → snake_case mapping).

## 3. Question queue screen (mockup 1b)

New route `src/app/app/series/[id]/queue/page.tsx` (+ client components):

- Header: back chevron, "Question queue", Edit.
- List of `pending` questions ordered by `position`:
  - Position 0 renders the **"Next up"** pinned treatment (accent border + chip).
  - Provenance line: `saved during Session N · <relative date>` (via `source_interview_id`) or `queued by <name> · <relative date>` (member adds).
  - Drag to reorder (persists positions); ⋮ menu: **Pin as next up**, **Remove** (status → `removed`).
- Small **"Add question"** input (member-visible, `source = 'member'`).
- **"Answer these now"** button → starts a talk with `mode = 'quickfire'` seeded from the queue (same interview start path, with a mode override param).
- Access: members with interview access can view and add; admins can reorder/remove/pin.

### Queue API

- `GET /api/series/[id]/queue` — pending items with provenance.
- `POST /api/series/[id]/queue` — add (`text`, `source`, optional `source_interview_id`).
- `PATCH /api/series/[id]/queue` — reorder (array of ids → positions), pin, remove, mark asked.
- `POST /api/interviews/[id]/queue` — convenience endpoint used mid-session by Flow's **+** button; authorized by interview access; stamps `source_interview_id`.

## 4. Mode runtimes

All modes flow through `src/app/api/interviews/[id]/realtime-token/route.ts` → `buildInterviewerInstructions()` in `src/server/ai/interviewer-prompt.ts`. The token route now reads `interviews.mode`, loads pending `queued_questions`, and attaches a per-mode `tools` array to the realtime session. Instructions remain rebuilt fresh on every mint (reconnects see current queue state for free).

### Deep dive

Today's prompt path, unchanged, driven by the stored legacy depth register (default `balanced`).

### Quickfire

- Instructions include a numbered question list: pending queue (by position) first, then must-cover topics ordered lowest-coverage-first, capped to what fits the session length.
- Behavioral contract in the prompt: ask each question nearly verbatim, one at a time; brief warm acknowledgment; **no** follow-up mining; move to the next; end when the list or time runs out.
- Tool `mark_question_asked({ queueId?: string, index: number })` — the model calls it after finishing each question. Client updates a progress indicator ("3 of 7") and PATCHes queue rows to `asked` (with `asked_in_interview_id`). Tool failures are non-fatal: the session continues; unmarked rows stay `pending`.

### Flow

- Tool `propose_followups({ questions: string[] })` (2–3 items). Prompt contract: after the subject's answer completes, call the tool and **stay silent** until the tool result arrives.
- Client (`LiveInterview.tsx`) on the function-call event renders the **"Where next?"** card stack (mockup 2b):
  - First card **Answer ›** — sends `function_call_output` `{ chosen: "<question>" }` + `response.create`; the model asks it aloud and the conversation resumes (2a).
  - Other cards **+** — `POST /api/interviews/[id]/queue`; card gets strikethrough/check, toast "Saved for later — Queue · N" (2c). Failure: error toast, card stays tappable.
  - Any card is answerable; + is available on non-selected cards.
- Footer during cards: **Queue · N** chip (opens a read-only queue sheet), center orb, **Stop here** (ends session → wrap).
- **Fallback:** if no `propose_followups` call arrives within ~4s of the user's turn ending, the client sends a `response.create` nudge instructing the model to call it; if it still doesn't, the session just continues conversationally (degrades to Deep dive, never blocks).
- Flow's opening question: the queue's "Next up" item when one exists, else the normal opening prompt.

### Pre-talk mode picker

When `ask_mode_each_time` is true, the Talk entry points route to a lightweight chooser (three cards with the same helper text as settings) before mic/WebRTC setup; the choice is written to `interviews.mode`. When false, sessions start directly in `conversation_mode`. The "Answer these now" button bypasses the picker (explicit quickfire intent).

## 5. Session wrap (mockup 2d)

Extend the existing end-of-session/completion flow rather than building a new screen:

- Header line: mode · duration · series title.
- Stats and "Captured" highlights only from data that exists at wrap time (post-processing is async — show what's ready, don't fabricate; link to the series for the rest).
- **Queue banner** (all modes, whenever pending > 0): "N questions in your queue — they'll open the next session. Review ›" → queue screen.

## Error handling summary

- Model never calls the Flow tool → nudge once, then degrade gracefully to normal conversation.
- Queue write fails mid-session → toast, card remains, session unaffected.
- `mark_question_asked` missed → rows stay `pending`; next Quickfire re-asks (acceptable).
- Reconnect mid-Flow → token re-mint rebuilds instructions including current queue; card state is client-local and resets to listening.

## Testing

- Unit: prompt builder snapshots per mode (deep/flow/quickfire, with/without queue items, legacy depth values); queue API auth + reorder logic; migration backfill mapping.
- Component: settings form diff-PATCH with new fields; queue screen reorder/pin/remove; Flow card stack states (render, +, answer, toast, fallback).
- Manual voice QA (required — realtime behavior can't be fully automated): tool-call reliability in Flow, quickfire pacing, picker → mode threading, wrap screen.

## Out of scope

- Session-length value changes (5/15/Open from the mockup).
- In-session mode switching.
- Cross-series queues; queue items are per-series.
- Editing a queued question's text (remove + re-add covers it for now).
